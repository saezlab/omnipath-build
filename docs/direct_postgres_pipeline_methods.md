# OmniPath direct-to-PostgreSQL build pipeline

This document describes the OmniPath build pipeline at the conceptual level
used in a methods section. The pipeline starts from source-specific input
modules, preserves raw-source provenance, maps records to a common
ontology-backed representation, resolves evidence entities to canonical
identifiers, and derives query-oriented graph tables.

## Overview

The OmniPath Metabo database is generated by a modular build pipeline that
separates resource acquisition, resource-level normalization, cross-resource
integration, and serving-layer materialization. The pipeline is organized as a
sequence of explicit stages:

1. input module execution,
2. normalized `Entity` construction,
3. raw record snapshotting,
4. mapping to source-scoped evidence,
5. identifier resolver preparation,
6. entity and relation canonicalization,
7. derived table and index construction.

The central design principle is to keep source evidence separate from the
deduplicated graph. Source evidence records retain their original source,
dataset, row identifier, snapshot identifier, identifiers, annotations, and
relation context. The graph layer is built from this evidence in a later pass,
so canonicalization decisions can be inspected, with full lineage.

## 1. Input modules and dataset definitions

Source data enter the pipeline through `input` modules. Each compatible
module defines one or more datasets and provides the source-specific knowledge
needed to retrieve, parse, and interpret those datasets. At the ingestion
level, each external resource is represented declaratively as one or more
datasets. A dataset combines three components: a download specification
describing how the source is retrieved, a raw parser that converts source files
into record dictionaries, and a schema mapper that transforms those records
into the shared internal `Entity` representation.

This design allows heterogeneous upstream formats to be processed through a
common ingestion model. Input modules are therefore responsible not only for parsing source files, but also
for declaring how source-specific fields map onto shared identifier
namespaces, annotation terms, entity types, and membership roles.

## 2. Normalized `Entity` representation

Resource-level normalization is performed against an internal ontology-backed
schema. Each normalized
`Entity` consists of four core elements: an entity type, a set of typed
identifiers, a set of typed annotations, and optional membership relations to
other entities. This allows proteins, metabolites, interactions, complexes,
pathways, reactions, and ontology terms to be represented with
the same structural model.

Entity fields are encoded using controlled vocabulary terms, primarily from
the Proteomics Standards Initiative Molecular Interactions controlled
vocabulary, supplemented where needed with OmniPath-specific terms. As a
result, identifiers such as UniProt, ChEBI, HMDB, and PubChem, and annotations
such as biological role, interaction type, taxon, evidence, and pathway
membership, are normalized to explicit semantic terms during ingestion.


In the relational representation, the nested `Entity` model is decomposed into
tables for evidence entities, identifiers, annotations, and relations.

## 3. Raw record snapshotting

The preparse stage executes the raw parser for each selected dataset and stores
the emitted rows as a content-addressed snapshot. Each raw row is normalized
into a canonical representation and assigned a stable content hash. This hash
is used to identify whether a row is unchanged, newly added, or removed
relative to the previous accepted snapshot.

Stable raw row identifiers are preserved across snapshots. When a row with the
same content hash reappears in a later run, it keeps its previous row
identifier. New rows receive new stable identifiers. The snapshot therefore
provides both a complete current view of the raw dataset and a delta against
the previous accepted view.

The snapshot manifest records the source, dataset, snapshot identifier, row
counts, identifier ranges, duplicate statistics, delta counts, paths, and
timestamps. If a newly parsed dataset is unchanged from the previous accepted
snapshot, the pipeline reuses the existing accepted snapshot rather than
creating a redundant replacement. This stage provides the reproducible input
boundary for all downstream work.

## 4. Mapping to source-scoped evidence

The evidence ingest stage converts accepted raw snapshots into a relational
evidence model. During normal operation, only raw rows marked as newly added in
the snapshot delta are mapped and inserted. Rows that disappeared from the raw
snapshot are removed from the source-scoped evidence layer together with their
dependent resolution and relation links. When mapper or parser logic changes,
the selected source dataset can be refreshed from the latest accepted raw
snapshot so that all current rows are remapped.

Each raw row is passed to its input-module mapper. The mapper emits a
normalized `Entity` object, ontology term, or structured record. These records
are decomposed into source-scoped evidence entities, identifiers, relations,
annotations, and memberships. The original source, dataset, raw row identifier,
and snapshot identifier remain attached to each emitted evidence record.

Structured records preserve their internal organization. For example, an
interaction-like record is represented as evidence for its
participants, plus relation evidence describing the subject, object, predicate,
and relation category. Complexes, memberships, and annotation-derived
associations are represented in the same evidence framework.

Ontology datasets are loaded as controlled-vocabulary entities with associated
labels, definitions, synonyms, alternate identifiers, and namespaces.
Ontology-backed annotations on biological entities are converted into
association evidence, allowing annotations to be queried through the same
entity-relation graph as other biological statements.

## 5. Resolver preparation

Identifier resolution uses lookup tables built independently from the source
evidence. Protein resolver tables map source identifiers such as UniProt
accessions, Ensembl identifiers, Entrez identifiers, HGNC identifiers, gene
symbols, and UniProt entry names to primary UniProt accessions. Chemical
resolver tables map identifiers such as ChEBI, PubChem Compound, HMDB,
LipidMaps, SwissLipids, or structural identifiers to standard
InChIKeys.


## 6. Entity canonicalization

Canonicalization assigns source-scoped evidence entities to deduplicated graph
entities. For each evidence entity, the pipeline collects its identifiers and
queries the resolver lookup tables for accepted canonical candidates. Protein
evidence is canonicalized to UniProt identifiers. Chemical evidence
is canonicalized to standard InChIKeys. Controlled-vocabulary terms
are represented by their ontology accessions.

The pipeline records all accepted resolver candidates for each evidence entity.
Candidates are then ranked by confidence, with direct identifiers preferred
over stable reference mappings and weak name-like mappings. If exactly one
best-ranked candidate remains, the evidence entity is resolved to that
canonical graph entity.

If no accepted candidate exists, the evidence entity is retained as an
unresolved graph entity using a stable fallback identifier derived from the
evidence identifier set, entity type, and taxonomy context. If multiple equally
ranked candidates remain, the evidence is marked ambiguous and is also attached
to a fallback graph entity. 

The output of this stage is therefore both a graph entity assignment and an
audit trail: each evidence entity has a resolution status, candidate count, and
reason. This makes unresolved and ambiguous records explicit rather than
discarding them during canonicalization.

## 7. Relation canonicalization

After entity evidence has been assigned to graph entities, relation evidence is
materialized into deduplicated graph relations. Each source-scoped relation
provides a subject endpoint, predicate, object endpoint, relation category, and
source provenance. Endpoints are resolved through the entity canonicalization
results, or directly when the endpoint already represents a controlled
vocabulary entity.

Relations are deduplicated by subject entity, predicate, object entity, and
relation category. Each source evidence row remains linked to the deduplicated
relation it supports. Relation annotations are also linked through the evidence
layer, so source-specific statements can be recovered even when many evidence
rows support the same graph relation.

This stage turns source-local biological statements into a unified graph while
maintaining the many-to-one relationship between source evidence and canonical
relations.

## 8. Derived query layer

The derive stage prepares tables and indexes used for query performance and
faceted exploration. It creates secondary indexes on the core evidence,
resolver, entity, relation, and annotation tables. It also builds derived
tables that summarize graph structure and controlled-vocabulary metadata.

The entity relation count table stores the number of distinct graph relations
incident on each entity. The ontology term table denormalizes controlled
vocabulary entities into searchable records.

The pipeline also builds bitmap-backed facet tables. These support efficient
computation of scoped filter counts for annotation term, entity type, taxonomy, source, ontology,
predicate, relation source, and participant type. Because these tables are
derived from the current graph state, they can be rebuilt after source refresh
or canonicalization without changing the underlying evidence model.

## 9. Provenance and reproducibility

The final database contains three connected layers: accepted raw snapshots,
source-scoped evidence, and deduplicated graph records. Raw snapshots define
the exact parser input used for a run. Evidence tables preserve the mapped
interpretation of each raw row. Graph tables provide canonical entities and
relations for query and integration.

Every graph entity or relation can be traced back through evidence links to
its source, dataset, raw row identifier, and accepted snapshot. This makes the
pipeline incremental while preserving reproducibility: raw record hashes drive
change detection, source-row synchronization removes stale evidence, and
canonicalization can be rerun over selected scopes without repeating the full
input parsing stage.
